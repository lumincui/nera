# Interactive Spec

此 skill 定义如何与用户一起维护持续演进的架构 spec，同时将项目内容和助手协作规则分开。

## 核心规则

- 项目架构内容放在 `spec/` 下。
- 助手协作规则放在此 skill 中，不放入项目 spec。
- 项目 spec 记录使用中文。
- 不记录用户表达以外的内容。
- 有阶段性进展时，及时通过 git 提交。
- 维护好 `.gitignore`。
- 如果要补充推理、建议或方案，先在对话中提出；只有在用户接受或表达之后，才写入 `spec/`。
- 在过程中持续维护 `GOAL` 和 `PRINCIPLE` 记录。
- 如果用户补充或纠正了助手的动作，考虑是否需要更新此 skill，让后续合作更紧密高效。
- 当用户明确要求补充上下文时，`spec/` 可以包含必要的外部知识 reference。
- 当用户说“记录火花”时，含义是记录思维火花或后续可能扩展点，不要机械地创建名为“火花”的章节。
- 设计过程中会有很多头脑风暴，思维的火花值得记录。
- 不是第一性目标的火花不要过度展开。
- 后续工作中，助手需要参与思考并提出建议。
- 对有建设性的意见，助手需要及时向用户建议。
- 建设性建议先在对话中提出，经用户确认后再写入项目 spec。
- 做产品或竞品调研时，优先按能力维度归类。

## Spec 结构规则

- 将 `spec/` 视为多级树状结构。
- `spec/` 需要以最终代码实现的结构来组织。
- 代码架构取决于沟通结构；spec 的沟通结构会影响最终代码结构。
- 从顶层概念到细节逐步展开。
- “展开”意味着将对应节点分裂为目录。
- 当某个节点被展开时，将原有文件转换为目录下的 `README.md`，并在该目录下继续向下展开。
- 水平拓展时，在同一目录层级增加并列文档或并列目录。
- 每次更新 spec 前，先检查当前 `spec/` 结构。
- 每次更新 spec 后，检查树结构是否仍然平衡：
  - 避免把太多无关主题放进同一个文档。
  - 避免在父层级尚未清晰时过早创建很深的目录。
  - 同一抽象层级的并列主题，优先使用并列文档或并列目录。
  - 某个主题需要细节拆解时，将该主题展开为目录。

## 分离边界

项目 spec 可以包含：

- 用户表达过的产品或系统目标。
- 用户表达过的架构原则。
- 已确认的系统设计决策。
- 关于系统设计的待澄清问题。
- 系统设计的结构化拆分。

项目 spec 不应包含：

- 关于助手应该如何工作的指令。
- 关于如何维护文件的规则，除非用户明确要求这些规则成为项目 spec 的一部分。
- 应该放在此 skill 中的 spec 维护流程说明。
- 未经确认的助手假设或扩展方案。

项目 spec 可以在用户明确要求时包含：

- 必要的外部知识 reference。
- 与当前设计直接相关的官方文档链接和事实摘要。

## 更新流程

1. 检查当前 `spec/` 树。
2. 判断新的用户输入属于：
   - 项目内容，
   - 项目决策，
   - 项目原则，
   - 待澄清问题，
   - 或助手协作规则。
3. 当用户要求持久化助手协作规则时，将其写入此 skill。
4. 将项目架构内容写入 `spec/`。
5. 重新检查 `spec/` 树是否需要调整结构。
6. 总结修改了哪些文件，以及记录的内容类型。
7. 如果用户补充或纠正了助手的动作，判断这是否是稳定协作偏好；如果是，将其更新到此 skill。
